\section{Architecture: Concept}
In essence, a computer is simply a machine that takes in an input, processes it according to an instruction set, and then gives an output, similar to a mathematical function. This definition is abstract and simple, but the reality is much more specific and complicated, hence the inclusion of this section. Computers are very specific pieces of technology and only function initially when data and instructions are both provided directly to it; they can not compute based on independent "thought." [note: unless programmed] These data and instruction are stored in various ways on a computer, which will be explained later. For now, what needs to be known is that the computer stores them in a format called binary.

\begin{adjustwidth}{2em}{0em}
\subsection{Binary}
In our daily lives, we use a numeric system called decimal, which uses ten digits from $0-9$ to represent numbers. In contrast, binary is a numeric system that uses only two distinct digits to represent numbers: $0$ and $1$. The two number systems operate the same as each other; the only difference they have is the digits they use. To understand how to use binary, we must first understand the mathematical formulation behind the system. Base-$x$ systems are numerical systems that have $x$ digits where the position of them in a number represents a power of that base with the coefficient of the digit. The exponent value begins at $0$ in the rightmost position, and is incremented by $1$ as the position also changes by $1$ (toward the right). For example, in the base-$10$ system, the number $16$ can be expressed as $1 \cdot 10^1 + 6 \cdot 10^0$ and $256$ can be expressed as $2 \cdot 10^2 + 5 \cdot 10^1 + 6 \cdot 10^0$. In general, a number with $n$ digits written in a base-$x$ system, such as $abc \cdots z$, where each letter represents a digit, can be expressed as $a \cdot x^{n-1} + b \cdot x^{n-2} + c \cdot x^{n-3} \cdots z \cdot x^{0}$. With this understanding, the conversion between binary (base-$2$) to decimal (base-$10$) systems should be easier. For example, the binary number $1100$ can be expressed as $1 \cdot 2^{3} + 1 \cdot 2^{2} + 0 \cdot 2^{1} + 0 \cdot 2^{0}$, which evaluates to $12$ in decimal. Mathematical operations, like addition and multiplication, also apply to the binary system, but are not included the sake of brevity. \\\\
\end{adjustwidth}

Now that we understand binary, we can discuss why it is used and how data and instructions are stored using it in the computer. Binary is the simplest format for computers to use, as it is a system that contains just two possible values, and is why computers use binary and not, say, trinary (base-$3$). Binary is easily representable in the hardware (using transistors, which will be explained in a later section) of the computer. Each $0$ or $1$ unit is referred to as a bit, the smallest amount of information on a computer. A bit can be thought of as a switch, where each switch can be either off or on, mapping to the $0$ or $1$ of a bit, respectively. Now the question that arises is how computers are able to store instructions and data as simply $0$'s and $1$'s.

\begin{adjustwidth}{2em}{0em}
\subsection{Information Storage in Binary}
Data includes integers, strings (text), booleans (true/false values), and other data structures that can be seen throughout programming and real life. Integers are easiest to understand, as they can be directly converted to binary ($24 \rightarrow 11000$). Strings can not be directly converted to binary, so instead they use a table to map certain binary numbers to characters, symbols, and other data seen necessary by the encoding scheme. A commonly used table to map characters is the American Standard Code for Information Interchange, or ASCII (Figure 0.2), which uses $1$ byte (defined as the size of a character; usually $8$ bits) for each code [note: this is standard ascii; extended sets exist]. In this system, numbers from binary $00000000-01111111$, or decimal $0-127$, denote control characters (manages text formatting and printer instructions), the letters of the English alphabet, and some special characters (selection of ones common in programming, math, and formatting). For example, in ASCII, the binary number $01000000$ (decimal $64$) is the "@" special character, and the binary number $00001000$ (decimal $8$) is the "[BACKSPACE]" control character. Booleans are simpler to understand. Since booleans can only have two values, they can easily mapped to a single bit, where $0$ is false and $1$ is true. However, the smallest unit of information computers can access is typically a byte at a time, so in practice, booleans are usually $1$ byte. Other data types follow in this pattern, where ones that can logically be directly converted to binary do exactly that, while more abstract concepts are usually stored with some sort of encoding or mapping. \\

Instruction storage requires knowledge from later sections, so, for now, a more abstract definition will suffice. The overarching idea is that the instructions written in some programming language must be converted into binary. To do this, many compilers convert the code into an intermediate universal form, called Assembly, usually directly before generating binary code. There are different types of Assembly based on the hardware of a computer, but they are all similar in how they can be converted to binary code. Generally, the process works by mapping each instruction in Assembly to some sequence of bits in binary, which the computer can understand and execute using a few pieces of hardware that will be explained in a later section. Since almost all programming languages convert their code to Assembly, which is directly converted to binary, it can be seen how instructions can be stored in binary.
\\\\
\end{adjustwidth}

\begin{adjustwidth}{2em}{0em}
\subsection{Binary Manipulation}
Now we can store information binary, but we need a way to manipulate it in order to do significant computing. The way computers do this, in the most basic form, are through logic gates. Logic gates are the "building blocks" of all circuits and work by taking in one or more bits as input, doing an operation on it, and outputting a single bit. A logic gate can be represented conceptually as a truth table, which defines which inputs correspond to which outputs. The AND, OR, NOT, NAND, NOR, XOR, and XNOR gates are the 7 basic logic gates. The NOT gate, for example, takes in one bit and outputs the opposite of that bit ($0 \rightarrow 1$, $1 \rightarrow 0$). Explanation of each logic gate is not included for brevity, so instead there are truth tables included for the rest of the basic gates ().

\section{Architecture: Hardware}
To understand how computers process information, there are some main hardware components within the computer we must understand first: the central processing unit, memory, and buses. There are also other hardware components, like input/output devices, which will be explained briefly.

\begin{adjustwidth}{2em}{0em}
\subsection{Central Processing Unit}

The central processing unit (CPU) is the core processor of the computer, meaning it is what does the primary computation necessary to do the tasks required for proper function. It performs calculations, logical operations, and performs all other tasks necessary to execute a program. The CPU is made of 3 main parts: the arithmetic logic unit, control unit, and registers.
\begin{adjustwidth}{2em}{0em}
\subsubsection{Arithmetic Logic Unit}
The arithmetic logic unit (ALU) is the component of the CPU that handles all mathematical and logical operations on binary, just as the name suggests. It is used for basic mathematical operations, usually addition, subtraction, multiplication, and division. It also perform comparison operations, usually greater than, equal to, and less than, and not equal to. Its final major use is to apply logic gates on data. To generalize, it can be stated that the ALU performs bitwise operations. Bitwise operations are all operations that involve manipulating individual bits, which encompasses the previously stated operations and many others, like bit shifts [READ MORE].
\end{adjustwidth}

\begin{adjustwidth}{2em}{0em}
\subsubsection{Control Unit}
The control unit is the part of the CPU that performs its core functionality: computation. It does this by cycling through fetch, decode, and execute stages. The CPU starts with the fetch stage. In the fetch stage, the control unit retrieves the current instruction from the memory using the instruction pointer and stores the encoded instruction in the instruction register (). For now, the memory can be thought of as a type of hardware used by the computer to store information (in bits); it is essentially the computer equivalent of human memory. Once the CPU retrieves the instruction from the memory, the decode stage begins. The instruction retrieved from memory is stored in a condensed form that includes an opcode and operands, where the opcode specifies which operation to do, and the operands specify the inputs to the operation (). The opcodes and operands are then interpreted by the hardware of the computer and converted into control signals, allowing for the correct instruction to be performed during the final stage: execution. After the instruction is executed, the instruction pointer is incremented and the cycle continues. Within, or outside, of the control unit is the clock, which is used to synchronize and pace each step of the fetch, decode, and execute stages of the cycle in order to allow for other hardware components to work in parallel with the CPU, increasing the overall computer speed. The clock runs in cycles and the clock speed is measured in hertz (Hz), where 1 Hz is equivalent to one cycle per second. It is important to note that these cycles are independent of other hardware and that the cycles referred to in the fetch-decode-execute cycle are different from clock cycles; the pace is generated by the clock and other hardware simply follow the clock cycles to work in parallel, with the stages of the fetch-decode-execute cycle usually taking 1 clock cycle each. Modern CPUs have clock speeds that are on the magnitude of GHz (billions of cycles per second).
\end{adjustwidth}

\begin{adjustwidth}{2em}{0em}
\subsubsection{Registers} 
Registers are the fastest type of memory for computers. The registers are a set of memory locations close to the control unit and ALU to increase their speed, usually achieving speeds of 0 or 1 clock cycles. The sizes of each of the registers vary between computers but are usually 64 bits in modern CPUs and 32 bits in older CPUs from around the mid 1980s till the late 1990s and early 2000s (64 bit CPUs technically existed before this time frame but were only found in super computers, only becoming the standard afterwards). There are a many registers in a CPU, with the main types being general purpose registers and special purpose registers. The general purpose registers, as the name suggests, are used for general tasks that instructions perform, such as adding or subtracting values (the inputs and outputs of the performed operations are stored in the registers). In the x86 architecture, which is what will be used in this book, these types of registers are named using letters the A, B, C, D followed by an X and preceded by either R (to specify a 64 bit register), E (to specify a 32 bit register), or no letter (to specify a 16 bit register). Additionally, the 16 bit registers can be split into the higher (bits $8 \rightarrow 15$) and lower (bits $0 \rightarrow 7$) and referred to by replacing the X in the register name with H or L to specify the higher and lower bits respectively. For example, the register ECX would refer to one of the general purpose 32 bit registers, AX would refer to one of the 16 bit general purpose registers, and BH would refer to the higher 8 bits of one of the 16 bit general purpose registers. An important thing to keep in mind is that even with higher register sizes, smaller registers can still be referred to, such as using a 16 bit register on a 64 bit CPU. The CPU simply masks the other values stored in the register and reads the amount of data specified by the register specified in the instruction (when doing this type of operation, the lower bits of the register are used; for example bits $0 \rightarrow 31$ would be used when referring to the EAX register of a 64 bit CPU, whose full register size goes from bits $0 \rightarrow 63$). The special purpose registers are a set of registers that have purposes that are outside of doing tasks from user programs. These registers are mostly used for tasks that are critical to the control of the CPU and the main ones that need to be known are the instruction pointer (\_IP). The instruction pointer is a register that holds the location of the current instruction being executed by the control unit, allowing the computer to keep track of where in a program the CPU is, even if it exits the main part of the program to complete other tasks like functions (). The instruction register is the register that holds the instruction itself, allowing for it to be decoded and executed by the control unit [INSERT PROGRAM COUNTER].
\end{adjustwidth}
\end{adjustwidth}

\begin{adjustwidth}{2em}{0em}
\subsection{Memory} 
The memory, also called the primary memory, is the main component in computers that temporarily stores information. The primary form of memory in modern computers is the RAM (random access memory). Memory can be thought of as a long row of values. Each box has a memory address associated with it, starting from 0, usually written in hexadecimal format.
\begin{adjustwidth}{2em}{0em}
\subsubsection{Hexadecimal}
These memory addresses are usually seen in hexadecimal (base-16) format which is denoted with a "0x" prefix. Hexadecimal uses the characters A-F to represent the decimal numbers 10-16; the hexadecimal digits are composed of numbers 0-9 and characters A-F. For example, the 42 in binary be equivalent to 0x2A in hexadecimal. A special property of hexadecimal is that each digit in hexadecimal is able to store 4 digits of binary, making conversion between each other easy. For example, to convert the binary number 101000010110 to hexadecimal, first the number should be split into groups of four digits (appending 0s to the front if needed): 1010 - 0001 - 0110. Then, each set of four digits should be mapped to their hexadecimal counterpart: 1010 (binary) -> 10 (decimal) -> A (hexidecimal). Conversion to binary as an intermediate is not necessary, it is just show to help build intuition. Doing this to each group gives us A | 1 | 6. Then, we just join the numbers together to get the final binary number: 0xA16. To convert a hexadecimal number to binary, this process is simply reversed, with each digit mapped to its binary counterpart, and then joining each set of four binary digits together.
\end{adjustwidth}
\begin{adjustwidth}{2em}{0em}
\subsubsection{Hexadecimal Usage}
One thing to keep in mind in this section is that computers do not and can not physically use hexadecimal values, as the basic hardware is in binary [talk about trinary]. Hexadecimal is only used as a convenient "abstraction" for binary to make values shorter and easier to understand. Hexadecimal numbers in operating system contexts are usually appended with 0s such that it is the same number of bits as the CPU mode; for example 0xDEAD would become 0x000000000000DEAD for a 64 bit CPU. This is because the CPU mode is equivalent to the register size being used, and to reflect this, the values are written such that they are exactly what are written in the registers: one of the main contexts that hexadecimal numbers are used. For example, the 64 bit RIP register, the instruction pointer simply contains a 64 bit memory address which holds the machine code instruction to be executed by the CPU (that is why it is called a pointer, it simply "points" to the location where the information of interest is stored); if the machine code being executed by the CPU was located at 0xCAFEBABE, the RIP register would contain the hexadecimal value 0x00000000CAFEBABE (of course, this would be physically be stored in the register as binary). Although the finite size of the registers introduces a maximum value they are able to hold (0xFFFFFFFFFFFFFFFF for 64 bit registers) and therefore a maximum memory address, different techniques that will be discussed in the future, like paging, will allow for an increased memory range.
\end{adjustwidth}
The memory is used to store a multitude of things, such as machine code, data values, etc. Generally, anything that can be written in binary can be written into memory. If the memory contains continuous binary machine code starting at some memory address, the CPU is able to start execution from that address. This requires that the machine code is flat (does not contain a special file structure that must be parsed and set up), executable (contains native machine code the CPU can run), and has no dependencies on code or data that is not stored in the binary itself; all information necessary for execution must be stored in and referenced by the binary. If this is not the case, extra setup is required, as we will see in the future chapters with executing program files, such as ELFs. Before we can move onto the next hardware component, there are two last thing to learn about memory.
\begin{adjustwidth}{2em}{0em}
\subsubsection{Types of Memory Addresses}
The first type of memory address, and the one that is used in the lower level part of the operating system are physical addresses. Physical addresses are memory addresses that directly correlate to locations on the physical hardware. This means that the physical address 0xDEADBEEF would correlate to the same address 0xDEADBEEF in the memory; there is no abstraction. Virtual addresses are the second type of memory addresses. Virtual addressses allow each program using virtual addressing to use memory as if it was a single, contiguous segment, abstracting away physical addresses. This allows programs to be flexible, as they no longer need hard coded addresses to store information, or to do any other tasks they are made for. A piece of hardware, called the MMU, performs the mapping from virtual to physical addresses (once it is setup by the operating system), since virtual addresses are just simulated and still require real physical addresses to map to in order to read and write information.
\end{adjustwidth}
\begin{adjustwidth}{2em}{0em}
\subsubsection{Modes of Memory Addressing}
Memory addressing the the way by which programs address memory. There are 4 main modes of addressing that we will use to write programs in different levels of abstraction. The first mode is direct addressing, which involves explicitly referencing a memory address in the program, such as writing an assembly command with one of the parameters being a memory address. The second mode is indirect addressing, where the address fed to the program is a pointer to a memory location that contains the actual address of interest (this pointing is similar to the instruction pointer register). The third mode is indexed addressing, where the address is stored as a base and offset pair. The base contains the starting point of the memory segment, and the offset specifies the address offset in that segment which contains the memory address. This is done due to its efficiency when traversing contiguous data structures, like arrays, its parallels with how hardware devices usually address, and to prevent hard coded addressing. The fourth mode of addressing is relative, which is similar to indexed addressing, except the base is usually the start address of the program in memory or the program counter.
\end{adjustwidth}
\end{adjustwidth}

\begin{adjustwidth}{2em}{0em}
\subsection{Buses}
Data buses are what allow all of these devices to speak to each other. The main buses are the data bus, the address bus, and the control bus. The data bus is a bidirectional bus that actually transfers the data from different locations, such as from the memory to a CPU register. The address bus is a unidirectional bus, from the CPU to the memory, used to specify memory addresses from which another bus, like the data bus can begin to read data from. The final bus is the bidirectional control bus, between the CPU and other devices, used to time operations and actions between the different components to keep the execution in sync. \section{Programming: Concept}
Now we understand the context we are working it. Now, the natural question that arises is how do we, the programmers, control all of this? We do this through programming languages. We will use two in this book: assembly and C. These are usually the standard for creating operating systems, as assembly provides the low level control required for setup of the operating system and specific CPU instructions, and C provides an abstraction once said setup is done, so that programming the operating system becomes much easier. Other languages can be used, but they require either lots of setup when compiling or have features that are unable to be compiled so that they can run in an operating system environment. Of course, once the base of the operating system is done and all required functionality of the languages is implemented, these languages can be used. Assembly is able to be used since it is a language written in the human-readable versions of CPU instructions, called instructions or mnemonics,  rather than binary opcodes. This is where we will see things like the "mov", "add", and "jmp" mnemonics. C is able to be used since it is a language also very close to assembly, but it is able to abstract away all of the instructions themselves into different features which are commonly used in programs, such as loops, which are then able to be compiled into machine language. For C, some compiler setup is required such that the programs outputted do not have any external dependencies that would not be found in a standalone operating system environment. A program without these dependencies are called a freestanding program, and we must specify this as the output to the compiler. For the example operating system, and this book's examples, these compilers will be NASM for assembly and GCC for C, but they can be switched out for other compilers that offer the same functionality.

\section{Programming: Assembly}
Assembly is what will create the foundation of the operating system. It is the closest (human-readable) language that we are able to get to machine code, and is directly compiled to it. Specifically, we will be using x86 assembly in this book, since it is about targeting x86 architecture computers with legacy devices. Computer architectures refer to the type of CPU being used, defined by its design and ISA (instruction set). Since the CPU that we are targeting is an x86/x86\_64 chip, the assembly language that we use will correspond to it (x86\_64 is the 64 bit extension of the x86 chips, since they were originally made for 32 bit operation). Assembly languages come in different formats for syntax, called flavors in the assembly domain. The two main ones are Intel and AT\&T syntax, where the difference is that in Intel syntax, the operands are in a "destination, source" order, while in AT\&T syntax, it is "source, destination". Additionally, in AT\&T syntax, registers have a "\%" prefix and values have a "\$" prefix. There are are other differences, but these are the most obvious ones. This will explain the differences in the code that can be seen across examples from various sources. It is out of the scope of this book to teach all of x86 assembly. Therefore, you are encouraged learn the basics of assembly on your own. To start off, here are some sources where you can learn the fundamentals of assembly easily: [w3schools] [olivestem youtube].
